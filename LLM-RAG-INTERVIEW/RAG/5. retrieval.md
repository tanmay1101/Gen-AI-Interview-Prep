# Retrieval in RAG

## Why retrieval is the next step

After chunking, embeddings, and indexing, retrieval is the runtime engine that decides **what evidence** reaches the LLM.  
If retrieval is weak, generation quality drops even with a strong model.

---

## Production retrieval flow (industry pattern)

`query understanding -> metadata filters -> hybrid retrieval -> merge/dedup -> rerank -> parent expansion -> prompt assembly -> answer with citations`

### 1) Query understanding

Extract:

- user intent (incident cause, change correlation, procedure request, etc.)
- entities (`service`, `env`, `region`, incident IDs, time hints)
- confidence (if low, ask a clarifying question)

### 2) Hard metadata filters first

Apply strict scope filters before semantic search:

- `env`
- `service` / `resourceId`
- `time range`
- `region`
- `tenant`

Why: improves latency, relevance, and safety.

### 3) Hybrid retrieval

Run both:

- dense/vector retrieval (semantic)
- lexical/BM25 retrieval (exact identifiers and keywords)

For your AIOps use case, hybrid retrieval is usually mandatory due to IDs and error codes.

### 4) Merge and deduplicate

Merge candidates from multiple indexes/sources, then dedupe by stable keys:

- `source_path`
- `event_id`
- `trace_id` / `operation_Id`
- `timestamp`

### 5) Rerank

Rerank top candidates using a relevance model and keep a smaller high-quality set.

Typical pattern:

- retrieve 40-80
- rerank
- keep top 10-20

### 6) Parent-child context expansion

If retrieved item is a child chunk, expand to parent context:

- telemetry: trace/time window bundle
- docs: section-level context

This improves answer completeness without losing retrieval precision.

### 7) Prompt assembly (grounded)

Build prompt using:

- top evidence chunks
- citations metadata
- explicit instruction: separate evidence from hypothesis

Never pass irrelevant large context just because it is available.

### 8) Answer policy

- cite every key claim
- if evidence is insufficient, say so
- avoid speculative root-cause statements without source support

## Latency, accuracy, hallucination controls

### To reduce latency

- filters first
- parallel retrieval across relevant indices
- smaller rerank candidate windows
- cache frequent query patterns

### To improve accuracy

- hybrid retrieval
- reranking
- parent-child expansion
- continuous eval on incident-style gold queries

### To reduce hallucination

- strict grounding prompt
- citation-required output
- no-evidence fallback response
- evidence-vs-hypothesis separation in final answer

---

## Retrieval eval metrics (must-have)

- Recall@k
- nDCG / MRR
- citation correctness
- unsupported-claim rate
- P95 retrieval latency
- cost per query

Use these metrics to tune:

- filter policy
- top-k and rerank top_n
- parent expansion depth
- hybrid weighting
