# Embeddings

## What is an embedding?

An embedding is a way to convert text (or other data) into numbers (a vector) so machines can compare **meaning**, not only exact words.

- An embedding is a dense numeric vector, like `[0.02, -0.11, ...]`.
- Similar meaning leads to vectors that are close in vector space.
- Different meaning leads to vectors that are far apart.
- Embeddings are generated by an embedding model.

## How embeddings are used in RAG

RAG uses embeddings in two phases.

### 1) Indexing time (offline)

- Split documents into chunks.
- Create an embedding for each chunk.
- Store chunk text + embedding + metadata in a vector index/database.

Each chunk typically stores:

- `chunk_text`
- `embedding_vector`
- metadata such as `env`, `service`, `timestamp`, `source`, IDs.

### 2) Query time (online)

- User asks a question.
- Convert the question into an embedding (using the same embedding model family used for indexing).
- Search for nearest chunk vectors in the index.
- Retrieve top relevant chunks.
- Send those chunks to the LLM as grounding context.
- LLM generates an answer based on retrieved evidence.

## Why embeddings matter in RAG

Without embeddings, retrieval depends mostly on exact keyword matching.  
With embeddings, retrieval can find relevant text even when wording is different.

For incident-investigation use cases:

- embeddings help capture semantic intent (ÔÇťdependency caused latency spikeÔÇŁ),
- keyword/BM25 retrieval captures exact tokens (`trace_id`, `operation_Id`, incident IDs),
- so hybrid retrieval (dense + BM25) is usually strongest.

## One-line interview answer

ÔÇťEmbeddings are vector representations of text meaning; in RAG we embed both document chunks and the user query, retrieve nearest chunks in vector space, and pass those chunks to the LLM for grounded answers.ÔÇŁ

## Different embedding techniques used in RAG

- **Dense text embeddings (bi-encoder):** Standard vector retrieval for semantic similarity.
- **Domain-specific dense embeddings:** Models tuned for code, legal, medical, finance, etc.
- **Instruction-tuned / query-aware embeddings:** Improved query-document alignment for retrieval tasks.
- **Multilingual embeddings:** Single semantic space across multiple languages.
- **Late-interaction embeddings (token-level):** Higher precision via token-level matching, usually higher cost.
- **Sparse embeddings:** Learned sparse lexical vectors for exact-term sensitivity.
- **Hybrid retrieval embeddings:** Combine dense vectors with BM25/sparse signals.
- **Contextualized chunk embeddings:** Add chunk context/summary before embedding to reduce ambiguity.
- **Multi-vector embeddings per document:** Separate vectors for title, body, summary, or fields.
- **Hierarchical embeddings:** Embeddings at section/paragraph/chunk levels for parent-child retrieval.
- **Query-expansion embeddings:** Embed rewritten or expanded query variants and merge results.
- **Multimodal embeddings:** Shared embeddings for text with image/audio/video where needed.

## When to use which (quick table)

| Technique | Best used when |
|-----------|----------------|
| Dense text embeddings | Default semantic retrieval baseline |
| Domain-specific embeddings | Corpus has specialized language/jargon |
| Query-aware embeddings | User queries are short/ambiguous and ranking quality needs lift |
| Multilingual embeddings | Questions and documents span multiple languages |
| Late interaction | Precision is critical and extra latency/cost is acceptable |
| Sparse embeddings | Exact keywords/identifiers matter strongly |
| Hybrid dense + BM25/sparse | You need semantic + exact-match performance together |
| Contextualized chunk embeddings | Chunks lose meaning without surrounding document context |
| Multi-vector per doc | Different fields (title/body/metadata) have different retrieval behavior |
| Hierarchical embeddings | You need child precision + parent context expansion |
| Query expansion embeddings | Queries are vague, underspecified, or have many synonyms |
| Multimodal embeddings | Evidence includes images/audio/video plus text |
