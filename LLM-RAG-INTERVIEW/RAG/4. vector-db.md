# Vector DB vs Vector Store

## What is a Vector DB?

A **Vector Database** is a database optimized to store and search vectors (embeddings) efficiently using nearest-neighbor algorithms, plus filtering and indexing features.

It usually provides:

- fast similarity search at scale
- metadata filtering
- persistence, updates, deletes
- replication/scaling (product-dependent)

Examples: Pinecone, Weaviate, Milvus, Qdrant, and Postgres with `pgvector`.

## What is a Vector Store?

A **Vector Store** is a broader application-layer concept/interface for where vectors are kept and retrieved from.

It can be:

- a vector DB backend,
- an in-memory FAISS index,
- a framework wrapper in LangChain/LlamaIndex,
- or another vector-capable retrieval layer.

So, **vector DB is infrastructure**; **vector store is the app-facing storage/retrieval component** (which may use a vector DB under the hood).

## Difference in one line

- **Vector DB** = backend system built for vector operations.
- **Vector Store** = app-level abstraction/component used by your RAG pipeline.

## One use case for each

### Vector DB use case

Production incident-investigation RAG across millions of telemetry and knowledge chunks.

Why:

- needs durable storage,
- high-QPS similarity search,
- strict metadata filtering (`env`, `service`, `time`, IDs),
- multi-user reliability and scaling.

### Vector Store (lightweight) use case

Local prototype over a few PDFs and notes during experimentation.

Why:

- quick setup with minimal ops,
- local/in-memory retrieval is enough at small scale,
- easy iteration before moving to production vector DB.

## Industry-grade RAG: what to consider for vector DB, storage, and retrieval

For an industry-grade RAG system, treat vector infrastructure as a production search platform, not just “embed and query top-k”.

### 1) Define retrieval SLOs first

Set clear non-functional targets before implementation:

- P95/P99 retrieval latency
- Recall target (for example Recall@k on eval set)
- Query throughput / concurrent users
- Data freshness SLA (near real time vs hourly)
- Cost budget per 1k queries

These targets determine index type, rerank depth, and filtering strategy.

### 2) Schema and record design

Store each vector record with:

- stable `id` (deterministic chunk/event id)
- `vector`
- source text/snippet (for rerank and citations)
- metadata fields (`env`, `service`, `region`, `tenant`, `timestamp`, source IDs)
- version fields (`embedding_model`, `embedding_version`, `chunk_version`)

Production rules:

- explicit schema (avoid silent field drift)
- idempotent upserts
- soft-delete/tombstone strategy
- versioned migrations for re-embedding/re-indexing

### 3) Embedding strategy and governance

- Use the same embedding model family for indexing and query embedding.
- Redact PII/secrets before embedding.
- Embed only retrieval-relevant fields.
- Keep model/version in metadata for rollback and drift diagnosis.

### 4) Index configuration and ANN choices

- Choose ANN type by scale and latency goals (HNSW, IVF, exact fallback).
- Align distance metric with embedding model guidance.
- Index/filter metadata fields used in query filters.
- Plan memory footprint and compression/quantization where needed.

### 5) Ingestion and indexing operations

- Batch ingestion (not row-by-row writes).
- Separate heavy bulk load windows from online traffic where possible.
- Retry/DLQ/backpressure handling.
- Rebuild strategy and blue-green rollout for major index changes.

### 6) Retrieval architecture (quality-critical)

Use a staged retrieval flow:

1. Hard metadata filters first (`env`, service, time, tenant).
2. Hybrid retrieval (dense + BM25/sparse lexical).
3. Merge and deduplicate.
4. Rerank top candidates.
5. Optional child-to-parent expansion for context.
6. Send only final grounded evidence to the LLM.

### 7) Multi-tenancy, security, and compliance

- Strong tenant isolation strategy.
- RBAC per index/namespace/tenant.
- Encryption in transit and at rest.
- Audit logging for query/access.
- Regional/data residency controls when required.

### 8) Reliability and observability

- HA/replication with failure-domain awareness.
- Backup and restore testing (not just configuration).
- Monitor:
  - latency P50/P95/P99
  - retrieval quality proxy metrics
  - indexing lag
  - memory/disk saturation
  - filter selectivity and cache behavior

### 9) Evaluation and release gating

Do not ship without an eval set and acceptance gates:

- Recall@k
- ranking quality (MRR/nDCG)
- citation correctness
- latency/cost
- failure analysis by query type

Run A/B comparisons for:

- index strategy
- filter policy
- rerank depth
- embedding dimensions

### 10) Practical default architecture

For enterprise incident-analysis style RAG:

- source-aware indices (telemetry, changes, knowledge)
- hybrid retrieval enabled
- rerank enabled
- strict metadata filtering first
- parent-child context expansion
- model/versioned embeddings
- continuous eval and drift checks

## Target architecture + config values (AIOps Incident Investigator)

This section turns the guidance into an actionable baseline for your exact corpus shape (`telemetry`, `changes`, `knowledge`, optional `inventory`).

### A) Chosen stack

- **Primary vector/search backend:** Azure AI Search (hybrid retrieval)
- **Embedding model:** `text-embedding-3-large` (quality baseline)
- **App retriever abstraction:** LangChain or LlamaIndex vector store wrapper
- **Rerank stage:** semantic rerank over merged candidates

### B) Index layout

- `telemetry_index`
  - app logs + platform diagnostics
  - fields: `vector`, `chunk_text`, `timestamp`, `env`, `service`, `region`, `tenant`, `trace_id`, `operation_Id`, `resourceId`, `severity`
- `change_index`
  - activity/deploy/config/alerts/incidents
  - fields: `vector`, `chunk_text`, `timestamp`, `env`, `service`, `incident_id`, `release_id`, `commit_sha`, `resourceId`
- `knowledge_index`
  - runbooks, SLOs, KQL, SOP docs
  - fields: `vector`, `chunk_text`, `section_path`, `doc_version`, `env`, `service`
- `inventory_index` (optional / light)
  - mainly routing/filter context, not heavy generation context

### C) Recommended retrieval parameters (starting defaults)

- **Hard filters first:** `env`, `service/resource scope`, `time range`, `region`, `tenant`
- **Hybrid retrieval:** enabled on all primary indices
- **Candidate pool before rerank:** 40-80 combined results (cross-index merged)
- **Rerank output:** top 10-20 chunks
- **Context to LLM:** 8-16 chunks after dedupe and child->parent expansion
- **Time-window expansion for incidents:** +/- 15 to 30 minutes around incident anchor by default

### D) Indexing/search technique defaults

- **ANN technique:** HNSW (vector fields) for low-latency search
- **Lexical technique:** BM25/full-text in parallel (hybrid)
- **Merge/rank fusion:** backend-native hybrid fusion + rerank stage
- **Metadata filtering:** pre-applied for scope safety and latency reduction
- **Parent-child context:** enabled for telemetry bundles and knowledge sections

### E) Ingestion defaults

- Batch upserts with idempotent deterministic IDs.
- Redaction before embedding (PII/secrets stripped or masked).
- Schema validation gate before write (reject malformed payload).
- Version tags in each record:
  - `embedding_model`
  - `embedding_version`
  - `chunk_version`
  - `pipeline_version`

### F) SLO targets (initial)

- Retrieval P95 latency: <= 400-600 ms
- End-to-end answer P95 latency: <= 2.5-4.0 s (depending on LLM)
- Recall@20 target: >= 0.85 on gold incident query set
- Citation correctness: >= 0.95 on audited sample
- Unsupported-claim rate (hallucination proxy): <= 2-5%

### G) Rollout plan

1. Launch with baseline model and full dimensions.
2. Run eval matrix on reduced dimensions and rerank depth.
3. Move to canary traffic (10-20%).
4. Compare quality/cost/latency against baseline.
5. Promote only if recall + citation metrics hold.

### H) Fallback path (if Azure AI Search is not chosen)

- Pinecone hybrid index + rerank:
  - single hybrid index for most teams
  - metadata filters + rerank + parent expansion
- Weaviate/Qdrant:
  - explicit schema
  - payload/filter indexes
  - quantization at scale

## Indexing in RAG (concept, visualization, benefits, and techniques)

### What is indexing?

Indexing is the process of building fast lookup/search structures over your data so queries do not need to scan everything.

In RAG, indexing usually means creating:

- vector indexes for semantic similarity,
- lexical indexes for exact keyword matching,
- metadata indexes for fast filtering.

### Visual example (library analogy)

Imagine a library:

- books = your chunks/documents,
- user question = search query,
- librarian = retriever.

Without indexing, the librarian reads every book page-by-page.  
With indexing, the librarian uses:

1. a keyword catalog (lexical index),
2. a topic map (vector index),
3. shelf filters (`env`, `service`, `time`) as metadata indexes.

This is why indexed retrieval is fast and relevant in production RAG.

### How can we do indexing?

Typical RAG indexing pipeline:

1. Clean and chunk data.
2. Create embeddings.
3. Build index structures:
   - ANN/vector index (for semantic search),
   - lexical/inverted index (for exact terms),
   - metadata/payload index (for filters).
4. Upsert records (`id`, text, vector, metadata).
5. Query path uses filters first, then retrieval, then rerank.

### Benefits of indexing

- lower latency (fast retrieval),
- scalability to large corpora,
- better relevance and precision,
- lower query-time compute cost,
- predictable production performance.

### Why experts use indexing in industry

Real enterprise queries contain both:

- semantic intent (“why did checkout fail?”), and
- exact tokens (`trace_id`, incident ID, error codes, release IDs).

So experts use hybrid indexed retrieval:

`metadata filter -> vector + lexical retrieval -> merge/dedup -> rerank -> context expansion`

This pattern improves relevance, keeps latency under SLOs, and reduces hallucinations by improving grounding quality.

### Different indexing techniques/types

#### 1) Lexical/inverted index

- BM25/full-text indexing for exact matches.
- Best for codes, IDs, names, specialized terms.

#### 2) Vector index (ANN)

Common ANN methods:

- **HNSW**: strong speed/recall, higher memory.
- **IVF/IVFFlat**: more memory-efficient, tunable recall/latency.
- **Flat/Exact**: perfect recall, slower at large scale.

#### 3) Metadata/payload/scalar index

- Fast filtering on fields like `env`, `service`, `timestamp`, `tenant`.
- Essential for scope safety and speed.

#### 4) Hybrid indexing

- Combine lexical + vector indexes.
- Most common production RAG approach.

#### 5) Parent-child/hierarchical indexing

- Retrieve small child chunks for precision.
- Expand to parent for context completeness.

#### 6) Multi-index by source type

- Separate indexes for telemetry, changes, knowledge, etc.
- Enables per-source tuning and cleaner relevance behavior.

### One-line interview answer

“Indexing in RAG is building fast search structures over vectors, text, and metadata so the system can retrieve grounded evidence quickly and accurately at production scale.”
