# Generation and Grounding in RAG

## Why this step matters

Generation is where users see the output, but grounding decides whether that output is trustworthy.  
A strong retriever can still produce poor answers if prompt design and output controls are weak.

---

## Generation architecture (production pattern)

`system policy -> grounded context -> user question -> constrained output schema`

### 1) System policy block

Define strict behavior:

- use only retrieved evidence
- cite every important claim
- separate observed facts from hypotheses
- say “insufficient evidence” when support is missing

### 2) Grounded context block

Pass only curated top evidence from retrieval:

- deduplicated and reranked chunks
- parent-expanded context when needed
- metadata for citations (`source`, `timestamp`, IDs)

Avoid stuffing the model with all retrieved content.

### 3) User question block

Include original query and normalized query context:

- extracted entities
- selected filters/time window
- incident identifiers

### 4) Constrained output schema

Use structured output format:

- `summary`
- `observed_evidence`
- `likely_factors`
- `unknowns`
- `recommended_actions`
- `citations`
- `confidence`

This makes responses auditable and easier to validate.

---

## Prompt design for AIOps incident use case

### Required response sections

1. **What happened** (fact-only)
2. **What changed before incident**
3. **Likely contributing factors** (clearly marked as hypothesis)
4. **What is still unknown**
5. **Recommended next checks/actions**
6. **Citations**

### Citation format recommendation

For each key claim include:

- `source_type` (telemetry/change/knowledge)
- `source_path` or document ID
- `timestamp`
- key identifier (`trace_id`, `operation_Id`, `incident_id`, `release_id`)

---

## Hallucination guardrails

### 1) Evidence-only policy

No claim without supporting retrieved evidence.

### 2) No-evidence fallback

If retrieval confidence is low:

- return “insufficient evidence in retrieved context”
- ask a clarifying question or suggest narrower filters

### 3) Contradiction check

If evidence sources conflict:

- state conflict explicitly
- avoid definitive root-cause statement
- suggest next disambiguation step

### 4) Action safety

For remediation suggestions:

- prioritize read-only and low-risk checks first
- avoid irreversible actions unless evidence is strong and policy allows

---

## Confidence scoring strategy

Compute confidence from:

- retrieval score strength
- cross-source agreement
- citation coverage ratio
- contradiction presence

Example labels:

- **High:** strong evidence, multi-source agreement, complete citations
- **Medium:** partial evidence, some gaps
- **Low:** sparse evidence or conflicting signals

---

## Response templates (recommended)

### Incident RCA template

- Headline summary
- Timeline of evidence
- Potential cause chain
- Confidence and gaps
- Next diagnostics
- Citations

### Procedural guidance template

- Applicable runbook section
- Preconditions
- Step-by-step actions
- Validation checks
- Escalation criteria
- Citations

---

## Evaluation for generation quality

### Offline evaluation

- citation correctness
- unsupported claim rate
- contradiction handling quality
- faithfulness to retrieved context

### Online evaluation

- user feedback on usefulness
- escalation acceptance rate
- incident follow-up correctness
- time-to-first-useful-answer

## 30-second interview answer

“After retrieval, I enforce grounded generation with a strict policy prompt and structured output schema. The model must separate facts from hypotheses, attach citations for every key claim, and return an ‘insufficient evidence’ response when retrieval is weak. This is how I reduce hallucinations and make incident answers auditable in production.”
